# ADR-0021: Role-based access control

**Status**: Proposed — depends on ADR-0020
**Date**: 2026-05-11
**Deciders**: (leave blank — to be filled by team)

## Context

Tenet 11 (least privilege by default) commits Nordlys to role-based
access control where every operator has a maximum TLP clearance
level and a minimum viable privilege set. ADR-0020 specifies TLP
enforcement at the mesh level (between peers) and at the database
query layer, but defers operator-level enforcement to this ADR.

FIRST TLP v2.0 defines RED as "for the eyes and ears of individual
recipients only." Nordlys's mesh-level enforcement (ADR-0020)
delivers RED objects only to named peer organisations. Within a
peer organisation, however, every operator currently sees every
object the peer received — there is no intra-organisational access
control. This satisfies the inter-peer distribution rule but not
the individual-recipient requirement of TLP:RED.

Operator-level RBAC is therefore a precondition for any deployment
that handles AMBER or RED material through the mesh. Without it,
Nordlys cannot claim compliance with FIRST TLP v2.0 for those
distribution levels.

## Alternatives considered

- **TLP clearance per operator with flat role model.** Each
  operator account has a single attribute: maximum TLP clearance
  (CLEAR, GREEN, AMBER, RED). The API gateway and database query
  layer filter objects by the operator's clearance. No other
  permission distinctions. Simplest model; a single attribute
  gates all access. The weakness is that it conflates distribution
  sensitivity with functional authority — an operator with RED
  clearance can also administer extensions, manage identities,
  and perform any other action. Real SOC teams have analysts who
  need to see RED material but should not administer the node.

- **TLP clearance plus functional roles.** Each operator has a
  TLP clearance level (distribution axis) and a functional role
  (authority axis): e.g. viewer, analyst, administrator. The
  functional role gates actions (create events, manage
  extensions, administer identities); the TLP clearance gates
  visibility (which objects the operator can see). Two
  orthogonal dimensions, independently assignable. More complex
  to implement but models real SOC team structures accurately.

- **Attribute-based access control (ABAC).** Fine-grained policies
  combining operator attributes, object attributes, and context
  (time, source IP, etc.). Maximum flexibility. Rejected for v1.x
  because the policy language, the evaluation engine, and the
  audit trail for fine-grained decisions are a significant
  engineering and operational cost that is not justified by the
  current deployment scale.

## Open Questions

### Group A — identity and authentication

Operator identity is local to the node — operators authenticate
to their own node's API gateway and have no mesh-wide identity.
The node supports two coexisting authentication mechanisms:

- **OIDC** (primary for most deployments). The node acts as an
  OIDC Relying Party against the organisation's existing IdP
  (Entra ID, Keycloak, Okta, or similar). This provides SSO,
  MFA, and integration with the organisation's user lifecycle.
  Configurable per deployment.
- **Local credential store** (always available). Username/password
  (bcrypt-hashed) and API keys for service accounts (monitor
  sessions, SIEM integrations, CI/CD). Functions as the sole
  authentication mechanism in air-gapped OT environments and
  as fallback when the external IdP is unreachable.

Both mechanisms coexist: an organisation may have OIDC-authenticated
operators alongside locally-managed service accounts on the same
node. Every authenticated session — regardless of source — is
mapped to a local RBAC identity with a role and TLP clearance.

**Authorisation is configurable per deployment.** Two modes:

- **Local authorisation.** The node administrator assigns roles
  and TLP clearance directly. OIDC provides identity only; the
  node ignores IdP claims for authorisation decisions. Simplest
  model; node administrator retains full control.
- **Claims-based authorisation.** The node maps OIDC claims
  (groups, roles, custom attributes) to Nordlys roles and TLP
  clearance levels via a configurable claim-mapping policy. This
  allows the organisation's existing AD group structure to drive
  Nordlys access. The mapping is defined and audited locally —
  the node never trusts raw IdP claims without explicit mapping.

The choice between local and claims-based authorisation is a
per-deployment configuration, not an architectural decision. Both
modes produce the same local RBAC identity and are indistinguishable
to the rest of the system.

Open questions:

- How does operator identity interact with the audit log? Every
  audit entry must attribute actions to a specific operator
  (including whether they authenticated via OIDC or local store),
  not just to the peer.
- What is the API key format and lifecycle? Rotation policy,
  scoping (per-endpoint or global), expiry?
- How are local accounts bootstrapped? The first administrator
  must be created before OIDC is configured — is this a CLI
  command, an environment variable, or a setup wizard?
- Should claims-based authorisation support a "ceiling" — e.g.
  the claim mapping can grant up to AMBER clearance, but RED
  clearance always requires explicit local assignment by a node
  administrator?
- When the external IdP is unreachable, should OIDC-authenticated
  sessions that are still valid (unexpired tokens) continue to
  work, or should the node require re-authentication through the
  local fallback?

### Group B — role model

- Is the functional role model fixed (viewer/analyst/admin) or
  configurable per deployment?
- Can a single operator hold multiple roles? Or is one role per
  operator sufficient?
- Who assigns roles and TLP clearance? A local administrator? Is
  there a bootstrap problem (who is the first administrator)?

### Group C — emergency escalation (Tenet 11 vs Tenet 6)

- What does the break-glass procedure look like? Time-limited
  privilege elevation with automatic expiry and immutable audit
  record — but what is the mechanism? A signed escalation token?
  A secondary authentication factor?
- Can an on-call operator self-escalate, or must another
  administrator grant the escalation?
- What is the maximum escalation duration? Is it configurable per
  deployment?

### Group D — extension interaction

- Do extensions have their own RBAC identity, separate from the
  operator who enabled them? (Expected answer: yes — an extension
  runs with the permissions declared in its manifest, not with
  the enabling operator's permissions.)
- How does operator clearance interact with extension clearance?
  An operator with GREEN clearance enables an extension declared
  as AMBER-capable — does the extension see AMBER objects? (Tenet
  10 says the extension's declared scope is enforced independently
  of the operator.)

### Group E — anchored to ADR-0020

This ADR cannot be closed until ADR-0020's TLP enforcement
architecture is finalised. Specifically:

- The TLP enum and rank ordering.
- The database query layer's mandatory TLP filter (which this ADR
  extends with per-operator clearance).
- The API gateway's SSE filtering (which this ADR extends with
  per-operator clearance).

### Group F — unattended display sessions (SOC monitor)

- Nordlys has two consumption modes: interactive (operator at desk,
  authenticated, personal clearance) and passive display (always-on
  monitor on SOC wall, no operator actively logged in). How is the
  passive display authenticated and what clearance does it hold?
- A wall monitor is visible to anyone in the room — visitors,
  cleaning staff, non-cleared personnel. Its effective clearance
  should arguably be CLEAR or GREEN at most. Is this a "device
  session" with a fixed clearance, a service account, or something
  else in the RBAC model?
- Should the display mode show reduced information (metadata only —
  timestamp, severity, source — without event body) even for GREEN
  objects, to limit exposure in a physically uncontrolled viewing
  environment?
- How do existing SOC operations handle this today? (Requires input
  from SOC operators at Statnett, Hafslund, or similar.) Common
  patterns in the industry include: dedicated read-only service
  accounts with minimum clearance, physically separate "unclassified
  dashboards", or the same UI with a display-mode toggle that strips
  detail.
- Does a monitor session need a different refresh/expiry model than
  an interactive session? (No human present to re-authenticate; the
  session must persist for days/weeks without intervention.)
- If a RED event arrives while a GREEN-cleared monitor is active,
  the monitor sees nothing — correct by design. But should the
  monitor indicate "N events exist above your clearance" (leaks
  count) or show nothing at all (no indication that restricted
  events exist)?
